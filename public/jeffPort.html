Bespoke Offers

About the project:

Bespoke Offers was a large-scale Daily Deals site in the UK by Barclaycard. It was a platform for providing products and services offers using various relevancy engines and harnessing Barclays’ extensive consumer database. It consisted of a consumer e-commerce application, a self-serve merchant site, and administrative sites for managing merchants and customers.

My role:

I was a Front End Developer on an agile development team, but I wore many hats for this position: Designer, Front End coder, Ruby and JS programmer, and of course QA analyst.

I employed the use of a stack of tools from Adobe CS6 to HTML5, CSS3 and Javascript and on into backend technologies like Ruby on Rails, rspec and cucumber for testing.

As a Designer...

Designs from vendor firms invariably required modifications and often didn’t account for many screensize breakpoints. Other features required the shorter turnaround of in-house design work. My duties included creation of desktop and mobile UX (Visio), page and feature layouts (Photoshop), SVG icons (Illustrator), etc. all within the confines of the Barclays and Bespoke style guides.

As a Front End Developer...

I wrote clean, semantic HTML and CSS using the Sass compiler using the mobile first approach. Among other features, I added the Add to Basket button functionality which opened a Added to Basket confirmation lightbox.

For user interactions such as, lightboxes and redirects, I worked with a variety of JavaScript libraries including jQuery, Angular, ZURB Foundation, and Bootstrap.

Challenges

Being part of such a large organization, there were many stakeholders whom often had requirements that negatively affected converegence. Note the sidebar of the Deal Page and, specifically, the abundance of information which requires the user to scroll in order to view it.

I utilized Adobe’s DTM Test & Target for user testing to give us solid data to back up these concerns.

As a Front End Developer...

One of the challenges I faced on the basket page was the variety of different features that could be added to any given offer.

For example, if an offer were for a product, it might have several variants, or if a product was only available for shipping, then S&H (P&P in UK) might be applied to the total cost. All of these different features had to be accounted for in limited real estate, and it of course had to be mobile responsive. I spent many hours of my life making that work.

I also assisted in the ever evolving design of the deal boxes within the slider and used JavaScript to implement the Top Offers slider carousel and infinite scroll for other relevant offer displays.

As a Ruby Programmer...

Most of what I did in Ruby on Rails was writing models, helpers, and tests. We were a Test Driven Development team, so as we wrote view code, we were expected to write tests concurrently.

Controllers, data tables and other more complicated back-end programming was handled by Engineers. Though I learned some of this on the job, and I am working towards becoming a full stack developer, I would not yet consider myself truly worthy of the “full stack” title.

Beat My Price

Beat My Price was a feature of Bespoke offers that allowed consumers to obtain optimal price points from services such as Amazon.

As a Front End Developer...

Beat My Price is an example of a feature which I did not take part in the design or initial coding, but rather that is something where I was required to blindly go into the feature’s code an fix defects. We used Pivotal Tracker to log defects.

My team at the time was responsible for Offer Relevancy. We made a personalized home page and personalized offer sections throughout the app. I was the lead designer and developer for this project. The homepage had a User profile with dynamically updated offer categories labeled as “My Interests”.

It allowed the user to select from a list of offer categories, which would then populate their home page. I implemented the use of infinite scroll on the desktop homepage and used a slider on other pages and mobile homepage.

As a Front End Developer...

I was often responsible for building out forms in the markup, linking the intake fields to a JavaScript file which would aggregate the data for Ruby controllers to update the database and generate an offer to load up to the Consumer site. I also had to find solutions for displaying the information on the Merchant side.

Challenges

As previously discussed, one of the major challenges of the Bespoke Offers project was the sheer volume of data. This is data that was already available, but also information we needed to collect and display.

Pretty dry stuff here and I never looked forward to working in the Merchant Portal because the code base for it was convoluted as a result of years of feature creep. Also, as UX/UI designers, we were limited in terms of any sort of experimentation.

That said, it was a vital part of the app, and it just had to be done. So, I suppose I’m adding this to the portfolio to show that I am capable of doing work which I am not in love with.
